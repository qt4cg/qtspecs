XQuery declare function with unprefixed name

In PR #2200 we changed the rules so that an XQuery function declaration where the function name is unprefixed treats the name as being in no namespace.

I would like to propose a tweak to this rule that improves backwards compatibility. Specifically:

  • If there is an explicit declare default function namespace declaration, then the existing (3.1) semantics are retained: an unprefixed name declares a function in that namespace.

  • The new rule kicks in only if there is no explicit declare default function namespace declaration. In 3.1, using an unprefixed function name with no explicit declare default function namespace would normally be a static error, because the "default default" is normally the FN namespace, which is reserved.

Replace `a/get(XX)` with `a/?(XX)`

The get(Expr) syntax in NodeTests feels clumsy - it looks too much like a function call. I would like to explore replacing it with ?(Expr) (with the parentheses optional as with lookup expressions).

For example, $map/get("date of birth") would become $map/?"date of birth", and $map/get($name) would become $map/?$name. It could also be used for selection by name in XNode trees: $parent/?$name would mean the same as $parent/*[node-name()=$name].

Achieving this isn't completely straightforward. The expression $map/?"date of birth" already matches the grammar, even in 3.1 - though in 3.1 it will always raise a type error, because the LHS of '/' must be of type node(), while the context item for unary '?' must be a map or array. In 4.0 the expression $map/?"date of birth" is perfectly valid, but it returns the same as $map?"date of birth" -- that is the actual date of birth, not a JNode that can be used for onward navigation.

One approach to delivering the revised semantics would be to make "/?" a two-character token, an operator in its own right.

get() can also appear after an axis, for example $map/descendant::get("date of birth"), or after "//" as in $map//get("date of birth"). I think we can make ? work in both these contexts, though it might be a little tortuous.

The new uses of ? would be followed by a KeySpecifier that follows the same rules as in lookup expressions.

xsl:analyze-string and lookahead/lookbehind

The XSLT instruction xsl:analyze-string is defined (a little informally) as post-processing the output of the fn:analyze-string function.

The output of fn:analyze-string has been extended to handle captured groups occurring within lookahead, and the spec of xsl:analyze-string needs to change accordingly.

In fact the current spec is rather vague: it doesn't really say how the groups delivered through regex-group() relate to the output of fn:analyze-string(): we need to tackle that at the same time. Potentially regex-group() needs to deliver additional information - not just the string value of the captured group, but its start and end position within the input string.

fn:unparsed-text, bin:decode-string: Encoding, BOM handling

When trying to write a PR for #2217, I noticed some general inconsistencies that we should clarify first.

Unifications

fn:unparsed-text and bin:decode-string use different rules for handling the input encoding:

Rule | fn:unparsed-text | bin:decode-string -- | -- | -- A. Require implementations to support… | utf-8, utf-16 | utf-8, utf-16, utf-16be, utf-16le B. Handling of initial xFF and xFE (before decoding) | ignored | interpret utf-16 as utf-16le C. Handling of initial \uFEFF (after decoding) | discarded | adopted (or interpreted, see PR for #2217) D. Implementation-defined heuristics to detect the encoding… | allowed | not allowed

I think we should…

  A. require implementations to support utf-8, utf-16, utf-16be, and utf-16le in both functions;   B. always interpret BOMs;   C. adopt (ZWNBSP) as Unicode input (after all, it should hardly ever occur at this place);   D. keep the current behavior unchanged (due to bin:decode-string being a more low-level operation)

BOM handling

As insinuated in B., I would suggest always interpreting a BOM if it exists, and skipping the bytes. If an encoding is specified that conflicts with the inferred encoding, we could…

  1. overwrite the inferred encoding,
  2. ignore the encoding argument, or
  3. raise an error.

Option 1 would be my preference, in adherence to the principle that user input should always be taken serious: If the input cannot be decoded with a specified encoding, this already results in an error (err:FOUT1200, most probably, although this needs to be clarified in the specs as well).
