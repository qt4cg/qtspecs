Make all data types ordered

No ordering is defined for the types QName, NOTATION, gY, gYM, gM, gMD, gD.

There seems no good reason for this restriction; it's perfectly easy to define an intuitive ordering for all these types (handliing timezones as we do for xs:date etc). In the case of QNames we sort first by URI then by local-name.

I propose that we remove the restriction.

This would make the types eligible for inclusion in functions such as sort, min, max, highest, lowest, compare, xsl:merge etc

With this, we can change the table for value comparisons so that instead of two separate supporting functions (e.g. op:boolean-equal and op:boolean-less-than) we have a single supporting function op:boolean-compare.

XQFO xref from xml-to-json() to element-to-map()

Many readers (or at least this one), wishing to convert an arbitrary XML fragment to JSON will first stumble across xml-to-json(), and won't know where to turn next when they realize that isn't quite what they wanted.

I believe there should be a note attached at the very top of the function description, ideally within the summary, along the lines of: "For conversion of arbitrary XML trees to JSON, see element-to-map()."

I tried to do this myself, attaching a note to <fos:summary> in the function catalog. But the schema yelled at me, so I thought it best to start an issue, to see how best to set up a cross-reference structure.

We will need a similar top-level xref at the entry json-to-xml() pointing to map-to-element(), if/when it is created per #1962

Split XSLT streaming into a separate specification

As one might expect (see #2203), there is pushback against dropping streaming entirely in XSLT 4.0.

However, a useful step I think would be to move streaming into a separate specification document.

I think this would be useful for readers; it would put all the information in one place, it would make it much clearer that the feature is an optional addition to the spec, and it would mean that those who aren't interested in it wouldn't be distracted by it.

But on a project planning level, it would have another big benefit: it would reduce the risk that finishing the work on streaming could jeopardise the whole project. It would become quite feasible to declare XSLT 4.0 complete even though the work on defining streaming is unfinished. This could potentially speed up completion of XSLT 4.0 by a year. Removing this risk is particularly important as there are few people on the project who understand the area really well.

parse-html() - the include-template-content option

i propose that we drop this option and instead always behave as currently specified for include-template-content : true().

It's an unnecessary burden on implementors to require them to provide two different ways of doing this. And I can't see that always including the content creates any problems for users.

The current Note suggests that the way an implementation behaves might depend on what architectural approach it is adopting to process the HTML. That in turn suggests that to support both options it might need to adopt a choice of architectural approaches depending on the option chosen: in other words, that the processor needs to implement two completely different approaches to implementing the function. That would indeed be a serious burden.

If there really are implementation difficulties in providing one approach rather than the other, then I think the right answer is to make the choice implementation-defined rather than making it a user option.

2190 Drop binary input for parse-csv and parse-json

This PR reverts the change to parse-csv() and parse-json() allowing them to take binary input.

The change was made primarily for consistency with parse-xml() and parse-html(). However, those functions have a legitimate reason to accept binary input, because discerning the encoding may be intermingled with parsing in the case of input documents that use custom syntax to define their own encoding. This is not the case for CSV and JSON, where standard decoding tools may be used.

In addition, there was no good reason to add the option to parse-csv() and not to csv-to-arrays() or csv-to-xml().

I would be happy if we could get rid of usages of treat as. Could we use coercion instead?

(: currently :)
for $map as map(*) in M 
return ($map?N treat as function(*))
  ($map, X, Y, ...)

(: proposed :)
for $map as map(*) in M
let $fn as fn(*) := $map?N
return $fn($map, X, Y, ...)

The error code would change from XPDY0050 to XPTY0004.

Drop XSLT Streaming?

I would like to float the idea that we drop streaming from the XSLT 4.0 specification.

See also #675 and #2128

The main reason for this is that I don't think we have the resources to do it. I don't think we're likely to find anyone other than me who will step up to the task; and if I attempt it I'm not sure we will find people willing and able to check my work, which is essential because it's devilishly difficult to get the detail right. We have added a lot of new features to the 4.0 spec without applying any thought to their streamability. The introduction of JNodes probably means quite a few rules need to be extended or qualified.

With the demise of Abel Braaksma's Exselt, Saxon is the only streaming processor available. As far as I can tell, streaming is not widely used: I'm sure we would get a lot more support requests and bug reports if it were, because it's difficult to write streamable code and I know that our implementation is fairly fragile. If the CG were to define all the streamability rules for new 4.0 constructs, I'm not sure we would be able to justify the resources to implement them.

In principle we could deprecate the language constructs associated with streaming from the spec and say that the effect of these constructs is implementation-defined.

There is precedent for this with the XQuery Update Facility. This wasn't carried forward to 3.0 partly because of lack of volunteers to do the work, and partly because of differences of opinion about what direction the spec should take. Implementors have found their own ways around the problem.
