Record declarations without namespace

Thanks to #2200, user-defined functions can now be declared without namespaces:

declare function yes() {};
yes()

We should extend this principle to record declarations:

declare record coord(x, y);
coord(1, 3)

A simple adjustment might be to adapt the rules in 5.20.2 Constructor Functions for Named Record Types, by changing…

The function name is the QName of the named record declaration, expanded using the default type namespace rule. The resulting QName must be the same as the module namespace if the declaration appears in a library module, and in any event, it must be in some namespace.

…to “using the default function namespace rule”, and dropping the last sentence.

Another solution would be include the default element namespace when searching for functions with unprefixed function names.

Computed constructors: expanded QNames

In computed element and attribute constructors, we should allow prefixes in name expression strings:

element { 'Q{uri}prefix:local' } {}

(: …equivalent to :)
element { parse-QName('Q{uri}prefix:local') } {}
element { QName('uri', 'prefix:local') } {}

As a side note: Now that we have QName literals, we could question whether we still need fn:parse-QName.

Dynamic XPath Evaluation the functional way

The xsl:evaluate instruction in XSLT 3.0 has proved highly popular. I've been studying a workload that uses it extensively. The Saxon implementation uses caching to prevent repeated parsing of the same expression, and on this workload we are getting a very high hit rate on the cache - over 99%. However, computing the cache keys is expensive: in fact, because the expressions are all very simple, computing the cache key probably costs almost as much as re-parsing the expression every time it is used. (One reason for this is that the cache key includes the entire static namespace context).

I would like to propose a functional equivalent of xsl:evaluate. Specifically, a function

fn:compile-xpath($expr as xs:string)

which returns a function $f such that $f($params) evaluates the compiled XPath expression with supplied parameters. This eliminates the need for caching (or rather, it enables the user to do the caching intelligently) which should give a performance benefit.

The proposed design also makes dynamic XPath evaluation available in XPath and XQuery, and generally fits better into the new functional way of working.
