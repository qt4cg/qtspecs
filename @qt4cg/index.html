EBV: maps, arrays, QNames

One of my favorite topics (sorry for being persistent): I would like to update EBV slightly to reflect the semantics and usage patterns of current versions of the language.

Currently, an EBV can be computed if $data is an XML or JNode, so you can do…

  • $node[.]
  • boolean($map-or-array/.)
  • if($xnode1 and $xnode2) then ...
  • if($map | $array) then ...

…but you cannot do…

  • $map-or-array[.]
  • boolean($map-or-array)
  • if($map1 and $map2) then ...

I think it would be much more intuitive to directly allow boolean($map-or-array) and to return true if the map or array exists.

Similarly, I cannot see boolean($qname) causing any trouble.

Subtyping: node() vs (doc() | elem() | att() | text() | comment() | pi() | ns())

In the subtyping rules, the type node() should be equivalent to a choice type of the seven specific node kinds.

More generally, any abstract type should be equivalent to a choice of its subtypes. For example, item() should be equivalent to gnode() | anyAtomicType | function(*), and gnode() should be equivalent to jnode() | node().

Guaranteed streamability

The XSLT Streaming specification defines conditions under which a stylesheet is deemed to be "guaranteed streamable": see §3.1 Streaming Guarantees, and requires a streaming processor under these conditions to process the source document in streaming mode.

The specification also states that if the stylesheet is not "guaranteed streamable", but streaming is requested, then the processor must report a static error.

The latter requirement is very burdensome because it requires the streamability analysis to be done without any tree rewrites of the kind that would turn a non-streamable construct into a streamable one. This means, for example, that type inferencing must be done before any rewriting. Saxon attempts to meet the requirement, but there are almost certainly areas where it fails to do so. Saxon is also non-conformant in that the option to check for guaranteed (i.e. interoperable) streamability is off by default. The only benefit of the requirement for users is that they know that if one conformant streaming processor can handle the stylesheet, then every conformant streaming processor will be able to do so. But even if multiple conformant streaming processors were available, this benefit would be minor in proportion to the burden on implementors.

I propose dropping this requirement from the spec. It will still be true that a conformant streaming processor must always use streaming if the stylesheet adheres to the rules in the spec; it will no longer be the case that a conformant streaming processor is required to raise an error if it doesn't.

JNode types, node kind tests: Unifications

It’s always rewarding when syntax needs only to be learnt once, and I think the current JNodeType and NodeKindTest rules could be further aligned:

  • With the JNode type syntax, a wildcard can be used for the sequence type: jnode(a, *). We should either allow this for element (attribute, …) tests – for example, element(name, *) – or we should discard jnode(name, *) – jnode(name) is equivalent.
  • Similar as for element tests, we could allow unions in JNode types: jnode(a|b|'c d'):
JNodeType      ::=  'jnode' '(' ( JKeyTestUnion ( ',' JValueType )? )? ')'
JKeyTestUnion  ::=  JKeyTest ( '|' JKeyTest )*
JKeyTest       ::=  JRootSelector | NCName | Constant
JValueType     ::=  '*' | SequenceType

Proposed renamings in the grammar:

  • NodeKindTest → XNodeType
  • DocumentTest → DocumentType, etc.
  • AnyItemTest → AnyItemType (analogous to AnyMapType etc.)
  • AnyNodeKindTest → AnyNodeType
  • NamespaceNodeTest → NamespaceType (in analogy with the rules for documents, which does not contain Node)
  • Basically, all RegularItemType and XNodeType RHS rules should probably end with Type.

If we prefer the Test naming scheme, we should rename JNodeType to JNodeTest, etc.
