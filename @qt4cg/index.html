2740 Drop constructor functions from built-in records

This PR drops constructor functions for built-in record types. My rationale:

  1. We avoid a bunch of rarely-used functions that would mostly go unnoticed anyway. If keeps the fn: namespace clean.
  2. We don’t have to deal with a broken argument order (some records list optional fields before required ones).
  3. Records can be built with a simple map. If required, coercion is available to turn them into the record.

If we want to keep the constructor functions, we should add at least one example how to benefit from them.

Closes #2740

Destructuring let: records

The current spec says:

let ${ $a, $b as xs:integer, $local:c } := { "a": 2, "b": 4, "c": 6, "d": 8 }
return $a + $b + $local:c

…is expanded to:

let $temp := { "a": 2, "b": 4, "c": 6 }
let $a := map:get($temp, "a", ())
let $b as xs:integer := map:get($temp, "b", ())
let $local:c := map:get($temp, "c", ())
return $a + $b + $local:c

We should use the lookup operator or a dynamic function call, so that let ${unknown} results in an error.

A side note, the rules for arrays say:

… replaced by a LetValueBinding of the form let Ai as Ti := array:get(V, i)

The subsequent example uses array:get($temp, 1, ()).

Destructuring in function parameters

@line-o convinced me that destructuring function parameters would be a fine (and I assume rather simple) addition to the language

Since #37 / PR #2055, sequences, arrays, and maps can be destructured in let clauses. The same bindings would be just as useful in function parameters: callbacks passed to fn:fold-left, fn:while-do, etc. often carry multi-part state, which currently needs an extra unpacking step in the function body:

(: current syntax :)
fn($state, $line) { let $($count, $errors) := $state return ... }

(: proposed syntax :)
fn($($count, $errors), $line) { ... }

The semantics could simply follow the existing let binding rules.

1506 Type declarations: Constructor functions

[NEW] This PR originally introduced constructor functions for type declarations, restricted to enums and atomic types (see below). As it turned out, the feature already existed in the spec (which also explains the existing test case).

My confusion was caused by the fact that we closed #1506 without further notice, and that this was not mentioned in the Item Type Declarations section. Next, its note claims that item type names never need to be retained at execution time, which contradicts the constructor functions. The PR tries to complete the picture, without changing the scope of the feature:

  • The section on item type declarations points to the existing constructor function rules.
  • Annotations rules/arity overlap have been added, they seemed to be missing.
  • A note has been corrected.

→ And we definitely need more test cases (indicated, for now, with the »Tests Needed« label). I will do my best soon.

[OLD] This PR introduces constructor functions for type declarations – a feature for which, surprisingly, we already have a test case, itemTypeDecl-026… (we certainly need more)

I decided to restrict the creation of constructor functions to declarations of enums and atomic types, for the following reasons mainly:

  • We avoid different semantics for record and type ... as record declarations.
  • Functions (incl. maps & arrays) can’t be built from a value; it would basically be a simulation of instance of/treat as.
  • Unions and choices are non-trivial: for declare type n as (xs:double | xs:decimal), n(1.5) the result depends on the order in which the type was written, which is easy to get wrong.

XSLT: Visibility of item types and record types across packages

The xsl:item-type and xsl:record-type declarations have a visibility attribute defining whether they are local to a package or can be accessed from using packages. They are also included in the types of component that can be named in xsl:expose and xsl:accept. But other details have not been followed through:

  • They are not included in the definition of "component" in §3.4.5
  • Because item types cannot be overridden in a using package (for good reasons), much of the discussion of component binding in §3.4.5 does not apply.

In fact it's debatable whether named types should fall within the definition of "components" at all; they certainly don't need to inherit all the complexity. However, they probably do need to be within the scope of xsl:expose and xsl:accept, which gives them some of the properties of components.

Any constructor functión associated with a named record type has the same visibility as the record type; it can't be overridden in a using package. This needs to be stated.

In §3.4.7, the paragraph starting "Type names used in the interface of public components...." definitely needs revision.

Not directly relevant to this issue, but the final para of §3.4.7 "At execution time, the schema available for validating instance documents contains (at least) the union of the schema components imported into all constituent packages of the stylesheet." also needs revision.

2569 Rules for unprefixed function calls with explicit default function ns

After further reflection, I have chosen to have three states:

  • The no-namespace-first search exists for one simple reason: without any declaration, the default function namespace is the reserved fn: namespace, where users cannot declare functions.
  • This means that the search is the only way to make unprefixed calls to user-defined functions work.
  • Once a default function namespace is explicitly declared, that reason disappears: users can declare functions there.
  • I believe the declaration should simply mean what it says: unprefixed calls go to the declared namespace and are not silently redirected by some no-namespace function that happens to exist.

The implementation is straightforward, and the proposed solution goes hand in hand with #2257.

Closes #2569

Map constructors: poor diagnostics

The change to map constructors to allow an expression (returning a map or JNode) in place of a (keyword: value) pair can make it very difficult to produce good diagnostics for simple errors. For example, I accidentally wrote:

declare %public variable $lib:dateTimeNow as lib:dateTimeSupplier := fn(){{"date": current-date#0, "time", current-time#0}};

leading to the error message

XPTY0004 The required item type of the second entry in the map constructor is (map(*)|jnode())

The actual error is using "," in place of ":".

I'm wondering if it might be better to require the user to signal their intent when using an expression that returns map entries, for example in place of

{ "monday": "work",
  if ($weekends) {
     {"saturday": "football"}
  }
}

require:

{ "monday": "work",
  + :  if ($weekends) {
     {"saturday": "football"}
  }
}

CSV parsing: delimiters

The current CSV spec provides a field-delimiter and a row-delimiter option. My suggestion would be to:

1. Drop row-delimiter

  • Line endings (CR, CRLF) are already normalized to \n before rows are split.
  • No mainstream CSV dialect uses a non-newline record separator.
  • Python, Java Apache Commons, Go, R, … don’t let you change it at all when reading. It seems to be overwhelmingly for serialization.
  • Removing the option also simplifies the CV0002/CV0003 error conditions. It would also simplify #2698.

2. Rename field-delimiter to separator:

  • “Separator” is predominant across the spec (we have item-separator, decimal-separator, grouping-separator, …); there is no other -delimiter.
  • With row-delimiter gone, the field- prefix is redundant and can be dropped.
