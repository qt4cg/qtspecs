Use standard coercion rules for the name expression in a computed node constructor

Computed node constructors currently have custom rules for type coercion of the results of the expression used to compute the node name. The rules are almost equivalent to the coercion rules for a required type of (xs:QName | xs:string) in the case of elements and attributes, or xs:string in the case of namespaces and processing instructions. But they are slightly different, for example they disallow promotion from xs:anyURI. These distinctions serve no useful purpose and prevent implementations reusing common code.

Binary Module: `bin:count-ones` for counting bits (population)

The Binary Module comes with helpful bitwise operators like bin:and, but no way to count set bits (bin:length counts octets). A population count, as offered by Java’s Long.bitCount or SQL’s BIT_COUNT, seems like a simple addition to me. For example, it is needed to compute the Hamming distance(bin:count-ones(bin:xor($bin1, $bin2))).

Proposed signature:

bin:count-ones(
  $value  as (xs:hexBinary | xs:base64Binary)
) as xs:integer

An equivalent XPath solution is doomed to be slow and more verbose. – I’ll be happy to do the PR.

`fn:matching-segments`: empty input

The use cases may be (very) limited, but for consistency, an empty sequence should be interpreted in the same way as in all other regex functions and treated as a zero-length string. The following query…

analyze-string((), '(.*)')

…returns…

<analyze-string-result xmlns="http://www.w3.org/2005/xpath-functions">
  <match>
    <group nr="1"/>
  </match>
</analyze-string-result>

…which is why…

matching-segments((), '(.*)')

…should return

{
  "substring": "",
  "position": 1,
  "groups": { 1: { "group": "", "position": 1 } }
}

FLWOR, member/key/value clauses: allow sequences

The new 4.0 FLWOR syntax makes it much easier to iterate over the contents of arrays and maps:

for member $m in $array
return ...

for key $k value $v in $map
return ...

Convenient processing of sequences is a core feature of XPath, and I think we should also support it here, by allowing sequences of maps or arrays as input for for clauses with member, key and value keywords. It prevents users from writing code like:

where exists($array)
for member $m in $array
return ...

for $map in $maps
for key $k value $v in $map
return ...

From the implementation side, this seems to be fairly trivial. Obviously, an error will be raised if a sequence contains a combination of maps, arrays, and items of any other type.

Lax validation with a non-schema-aware processor

The conformance rules for XSLT suggest (not very clearly) that validation="lax" with a non-schema-aware processor should have the same effect as validation with an empty schema: effectively, no validation, because the elements being validated are not present in the schema. The rules for XQuery say clearly that a non-schema-aware processor should raise an error for a validate lax expression.

The inconsistency becomes apparent when you do doc("input.xml", {'schema-validation':'lax'}). We would like this to have the same effect whether used in XSLT or XQuery.

file:descendants: recursion, filtering

It is advisable for an implementation (even if this is not addressed in the spec) to evaluate file:descendants fully before returning results. Otherwiese, a file walker may become corrupt, for example when a directory is deleted while it is being traversed. On the other hand, this means that the filtering can only take place after the full result is available.

Other programming languages differentiate between filtering the directories to traverse and the results to return, and I think we should do the same. My proposal:

file:descendants(
  $path     as xs:string
  $options  as record(
    recurse  as (fn(xs:string, xs:integer) as xs:boolean?)? := true(),
    filter   as (fn(xs:string, xs:integer) as xs:boolean?)? := true()
  )? := {}
) as xs:string*

With recurse, the directories to be traversed can be selected; with filter, the actual results can be reduced. The xs:integer value indicates the current position; this way, traversal can for example be aborted after the first n directories.

Coercing a map to a record type

In the current coercion rules (rule 10 for coercing maps to a record type), the note includes the sentence

But if the supplied value is { "latitude": 53.2, "longitude": 0, "altitude": 3_500 }, then a type error occurs because the field "altitude" is not defined in the record type.

However, there is no rule supporting this. The rules as written cause "altitude" to be ignored.

I think that Christian made the argument that this should be an error because of the risk that "altitude" is a misspelling of one of the defined field names, and I accepted this argument.

Though I have mixed feelings about it.... If there are extraneous entries in the map that you want ignored, it's not easy to discard them. Also, on the current rules a record with fields (latitude, longitude, altitude) is a subtype of a record with fields (latitude, longitude) and should therefore be substitutable.

Perhaps the answer should be that record-to-record coercion works differently from map-to-record coercion.
