Pipelining file operations

I pointed out in #2408 that this example

'test.bin'
=> file:write-binary(xs:hexBinary('00'))
=> file:read-text(fallback := true())
=> string-to-codepoints()

doesn't work, because file:write-binary() returns an empty sequence.

This raises the question of whether there is any way of writing this that is guaranteed to perform the write-binary() before the read-text(), and I don't think there is. This affects all the file functions that are called for their side effects. Many implementations are likely to evaluate

(file:write-binary(), file:read-text())

sequentially, but if you rely on this you're at the mercy of the optimizer.

One solution would be to make these functions return the filename, which would make the above example work: but this would not be backwards compatible. A solution to that might be to define a variant of the function that does this (e.g. a function with a different name, or with an extra parameter).

Another solution might be to define a construct

do {X, Y, Z}

that is defined to evaluate X, Y, and Z in that order and return the result of Z.

Of course a pure functional solution would get into the area of monads.

`file:delete`: revise error handling

file:delete raises an error if the specified path does not exist. I propose changing this behavior to ignore non-existing targets silently, and it allows users to get rid of the frequent if(file:exists($path)) then file:delete($path) pattern.

This change introduces a minor backward incompatibility, but it simply avoids raising errors instead of introducing new ones. In addition, it would be symmetric to the already idempotent file:create-dir function.

We should add a note similar as for file:copy, which states that the use of the function is powerful and won’t allow rollbacks.

Function signature of fn:subsequence

I propose that we change the argument type of the second and third arguments of fn:subsequence from xs:double to xs:numeric.

These arguments are xs:double for legacy reasons; if we defined it today they would be xs:integer. In practice the supplied arguments will almost always be integers, except for some peculiar 1.0 constructs where it's handy to supply infinity or NaN. If the function is implemented as declared, then a supplied integer gets converted to a double and the implementation typically has to convert it back. But if the implementation declares the type as xs:numeric (as Saxon does), then in edge cases involving function subtyping it's non-conformant.

There is a minor incompatibility. In 1.0 compatibility mode, the call subsequence($s, 1, "INF") is currently accepted because the expected type is xs:double so "INF" gets treated as number("INF"); supplying a string isn't allowed if the type is declared as xs:numeric. However, Saxon has been using xs:numeric as the required type for years (I checked back to 9.6), and this has never been raised as a problem.

Update records: `with { ... }`

@michaelhkay proposed in https://github.com/qt4cg/qtspecs/issues/2275#issuecomment-3812384613 that we could introduce a dedicated operator to update maps:

I wonder if there's a case for an operator equivalent to map:merge with duplicates="use-last", so one could do (say)

geo:coord(x := 3, y:= 5) with {'z': 3} with {'y': 4}

This operator would allow us to update multiple values in one step:

geo:coord(x := 3, y:= 5) with { 'z': 3, 'y': 4 }

We could allow processors to reject updates if these would violate the type definition of the incoming record. As indicated in #2518, though, it will hardly be possible to achieve this statically.
