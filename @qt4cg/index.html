Range constraints in fn:dateTime-record fields

This follows on from https://github.com/qt4cg/qtspecs/issues/2263#issuecomment-3961934656 (by @michaelhkay)

We also need to say more about the constructor functions associated with built-in record types.

For example, consider fn:dateTime-record. Does the constructor function for this record type impose any constraints beyond those in the type signature? Does it ensure that month is in the range 1-12? Does it ensure that if month is 4, then day is in the range 1 to 30? Does it ensure that if hours and seconds are present, then minutes is also present?

And editorially, where do we document this?

At the moment F&O § 9.3 Record fn:dateTime-record reads as though non-optional range constraints are being specified for most of the fields.

If they're not optional, when are implementations expected to confirm that values conform to their respective field constraints? When accessed via fn:build-dateTime the spec says that:

A dynamic error is raised [err:FODT0006] if any of the components in $value is outside the permitted range (for example if the value of the minutes component is less than zero or greater than 59), or if the combination of fields is not a valid date/time (for example, if month is 4 and day is 31).

Is an implementation also expected to check the field-specific ranges when a value is retrieved using, for example, ?day or map:get($my-dateTime-record, 'day')?

Similarly, is an implementation expected to validate inputs when the record is initially constructed (via fn:parts-of-dateTime or the implicit constructor)? Or, also, each time they're modified, for example, using map:put?

For further context, when a user defines a custom record type in XSLT (xsl:record-type), it doesn't appear possible to specify a range constraint.

Rounding dates/times and durations

Continuing on from https://github.com/qt4cg/qtspecs/issues/2406

In F&0 § 23.2 it says that:

When casting from a string to a duration or time or dateTime, it is now specified that when there are more digits in the fractional seconds than the implementation is able to retain, excess digits are truncated. Rounding upwards (which could affect the number of minutes or hours in the value) is not permitted.

For consistency, shouldn't that rule apply generically, when reducing the precision of a dateTime or duration? To truncate rather than round.

[XSLT] Default priority of predicate patterns

We appear to have inadvertently changed the default priority of patterns such as .[. instance of array(*)] from +1 to +0.5.

The change appears to have been conscious because test match-131 has been updated to reflect the change. However, no backwards incompatibility is recorded in the spec, and I can't see the justification for introducing one. I suspect that when we (probably me!) made the change, we were under the impression that it did not affect copnstructs permitted in XSLT 3.0.
