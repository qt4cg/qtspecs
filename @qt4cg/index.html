map:keys() and map:keys-where()

In PR #515 it is recorded that we decided to merge map:keys and map:keys-where into a single function.

But they are separate functions in the current spec, so I'm not sure what happened - did we make a decision to revert this change, or was there some error in the implementation of the PR?

The subsequent PR #969 seems to assume that they are still separate functions.

Personally I'm happy with the status quo but I wanted to check.

I would also be quite happy to drop map:keys-where - I don't think it adds significant value over map:filter.

fn:unparsed-text, bin:decode-string: Encoding, BOM handling

When trying to write a PR for #2217, I noticed some general inconsistencies that we should clarify first.

Unifications

fn:unparsed-text and bin:decode-string use different rules for handling the input encoding:

Rule | fn:unparsed-text | bin:decode-string -- | -- | -- A. Require implementations to support… | utf-8, utf-16 | utf-8, utf-16, utf-16be, utf-16le B. Handling of initial xFF and xFE (before decoding) | ignored | interpret utf-16 as utf-16le C. Handling of initial \uFEFF (after decoding) | discarded | adopted (or interpreted, see PR for #2217) D. Implementation-defined heuristics to detect the encoding… | allowed | not allowed

I think we should…

  A. require implementations to support utf-8, utf-16, utf-16be, and utf-16le in both functions;   B. always interpret BOMs;   C. adopt (ZWNBSP) as Unicode input (after all, it should hardly ever occur at this place);   D. keep the current behavior unchanged (due to bin:decode-string being a more low-level operation)

BOM handling

As insinuated in B., I would suggest always interpreting a BOM if it exists, and skipping the bytes. If an encoding is specified that conflicts with the inferred encoding, we could…

  1. overwrite the inferred encoding,
  2. ignore the encoding argument, or
  3. raise an error.

Option 1 would be my preference, in adherence to the principle that user input should always be taken serious: If the input cannot be decoded with a specified encoding, this already results in an error (err:FOUT1200, most probably, although this needs to be clarified in the specs as well).
